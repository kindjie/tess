# Third Audit — 2026-07-11 (v1.0.0, performance focus)

Post-v1.0.0 full audit of tess at main `0f2260a` (tag v1.0.0).
Emphasis: performance. Method: five parallel specialized reviews
(pathfinding, storage/ECS/topology, scheduling/ops/sim/diagnostics,
benchmark infrastructure incl. a live release-build run, and an
adversarial correctness sweep) plus an independent cross-lab review
(Codex CLI), findings deduplicated and each load-bearing claim
verified against source or measurement before inclusion. Prior audits:
`audit-2026-07-09.md` (S1-S4); S5-S11 code had per-PR review only and
is audited holistically here for the first time.

Verdict up front: no correctness defect above LOW severity was found
across all 48 headers. Thread-safety contracts match implementations
(worker-pool join ordering verified sound), diagnostics compile to
true zero cost when disabled (two independent confirmations), release
bench builds use genuine `-O3 -DNDEBUG`, and every gated benchmark
passes. The findings below are almost entirely performance headroom,
plus one HIGH benchmark-integrity defect.

Severity legend: HIGH = act now; MED = real cost on a hot or scaling
path, schedule it; LOW = worth fixing opportunistically.

---

## H1. Five benchmarks are fully optimized away; their gates protect
nothing

- `bench/tess_bench.cc:393` `storage/field_span_acquisition`
- `bench/tess_bench.cc:403` `storage/chunk_field_write_read_iteration`
- `bench/tess_bench.cc:416` `storage/single_chunk_page_iteration`
- `bench/tess_bench.cc:430` `storage/flat_array_iteration`
- `bench/tess_bench.cc:583` `block/scratch_allocate_u32`

Measured on this machine (release preset): 257-283 million iterations
at 0.25-0.27 ns — a single empty-loop cycle — versus hundreds of ns of
real work (the loops write and sum 4096 tiles). Root cause verified in
source: the write/read benches store into a local page that is never
observed after the loop and fold `sum` to a compile-time constant, so
`DoNotOptimize(sum)` alone preserves nothing; the span/scratch benches
call `DoNotOptimize` on loop-invariant pointers hoisted out of the
loop. Consequences: the three-way storage layout comparison these
benches exist to make (ChunkPage vs SingleChunkPage vs flat array) is
currently meaningless, real regressions in those paths are invisible,
and a future compiler that de-elides will fire a false gate failure
(real work >> the 25 ns cap).

Fix: `benchmark::ClobberMemory()` after the write loops and/or
`DoNotOptimize(page)` inside the loop; then recalibrate the five caps
against real numbers. `diagnostics/record_timing` (0.27 ns,
`tess_diagnostics_bench.cc:42`) is borderline-suspicious and should be
checked with disassembly while in there.

## M1. `weighted_path_batch` rescans all requests per shared-goal group

`include/tess/path/detail/weighted_batch.h:385-406`. The goal-count
map is built with an open-addressed hash, but member scatter then
loops over all n requests once per multi-member group — O(groups x n),
quadratic worst case. Found independently by two reviewers. The
sibling code already contains the right pattern: counting-sort group
buckets (`group_offsets_`/`group_members_`) at
`include/tess/path/path_runtime.h:451-467`. Reuse it here.

## M2. Sparse residency fingerprint: 3x redundant probes per chunk,
recomputed per batch member

Two stacking costs, found independently by two reviewers:

- `include/tess/storage/sparse_world.h:239-255`
  `residency_fingerprint` calls `resident_slot(key)`,
  `residency_generation(key)`, and `meta(key)` per resident chunk —
  three directory probes for data available at zero probes by
  iterating resident slots directly.
- `weighted_distance_field_path` calls `residency_matches` (a full
  fingerprint recompute) once per request
  (`weighted_batch.h:197-198`), so a batch pays
  O(requests x resident_count) for one unchanging resident set. The
  header comment on the fingerprint already names per-batch hoisting
  as the natural fix.

Fix both: slot-direct iteration inside the fingerprint, and hoist one
fingerprint per batch/tick (or maintain an incrementally-updated
residency epoch). Related LOW: a 64-bit commutative fingerprint can in
principle collide and authorize stale slot-indexed reuse; a monotonic
per-world mutation epoch would be deterministic. Same pattern at
`include/tess/topology/topology.h:1390-1399` and `:1161-1166`
(`is_region_graph_fresh` / `update_region_graph`): two probes per
resident chunk per pathing-tick check; one `resident_slot` lookup can
feed both reads.

## M3. Shared-goal field builds flood the entire reachable component
(no early termination) — proposal, needs benching

`weighted_batch.h:73-155` (`build_bounded_weighted_distance_field`)
expands the goal-rooted Dijkstra until the frontier is exhausted, but
in batch use (`weighted_path_batch`, and the runtime's repeated-goal
path) the consumers are exactly the group's start tiles, which are
known up front. Terminating once all group starts are settled bounds
the flood to the agents' region instead of the whole open map. This
composes with the S11.4 soak observation (goal re-arm replans the
whole batch on open maps) and the recorded per-agent-dirty backlog
item. Not previously tried per `optimization-log.md`. Care: the
field-reuse contract (`has_goal_`/`residency_matches` consumers of a
retained field) must either keep full floods for the caching path or
record the settled frontier bound; batch-local scratch is the easy
first target.

## M4. Queued-ops planning allocates and scans per tick

`include/tess/ops/queued.h:615-646` materializes a fresh chunk-key
vector per ResidentChunks op and calls `dirty_chunks`/`active_chunks`,
each a full O(chunk_count) metadata scan returning a fresh vector;
`:764` builds a new `ExecutionReport` (per-op chunk vectors) per run.
Hazard detection (`:678-691`, phase scan `:887`) is O(ops^2 x chunks).
Acknowledged in part by `sim/auto_exec.h:60`. At current bench scale
(64 chunks) this is cheap; it scales linearly-to-quadratically with
world size and op count. Fix: reusable planning scratch/report,
domain ranges instead of materialized key vectors, and (if op counts
grow) chunk/field ownership tables for near-linear hazard checks.

## M5. Dense per-tick scans stream 80-byte `ChunkMeta` to read 4 bytes

`include/tess/storage/world.h:313-321` (`collect_matching_chunks`,
feeding `dirty_chunks`/`active_chunks`) walks every chunk's ~80-byte
meta to test one `uint32` flag word — about 20x the necessary
bandwidth, every maintenance/queued tick, regardless of how few chunks
are dirty. The same full-scan ceiling shows up in render/delta
collection (`sim/render_delta.h:87-107`, `sim/delta_frame.h:765-784`).
Two fixes compose: (a) split the hot flag words into a parallel SoA
column (or summary bitset) so the scan touches 4 bytes/chunk and
vectorizes; (b) shrink `ChunkMeta`
(`include/tess/storage/chunk_meta.h:16-25`) by moving the cold 48-byte
inline `Box3 dirty_bounds` to a parallel cold array (~80 -> ~24 bytes,
one cache line per meta touch). Longer-term (design-level): a
maintained dirty-chunk set would remove the O(chunk_count) tick floor
entirely.

## M6. `Schedule::run_tick` traverses all tasks once per phase; dirty
merges write every task

`include/tess/sim/schedule.h:238-246`: 11 full passes over `tasks_`
per tick (the measured ~517 ns floor for 100 no-op tasks is exactly
this). `:339-346` + `notify_dirty` (`:225`): each dirty-producing task
ORs its mask into every task, though only OnDirty cadences read
`pending_mask` — O(dirty-producers x tasks). Fix: partition tasks by
phase at `seal()` (IDs stay stable) and keep an OnDirty-only index or
a frame-level accumulated mask.

## M7. Movement commit re-resolves the same coordinates 4-7x per step

`include/tess/sim/movement.h:154-232`: one validated+committed step
resolves `intent.to` ~6x and `intent.from` ~4x (passability, occupancy
field, reservation field, version check, three commit writes, two
dirty marks — each `field<>()`/`meta()` re-runs the full
coord->chunk->local resolve). `movement_versions_match` (`:109-145`)
additionally resolves both endpoints and reads two metas even when the
intent carries no version expectations (the movement-scheduler path,
`sim/path_agent.h:233`, never sets them). This is per-agent per-step
per-tick. Fix: resolve endpoints once into `ResolvedTile` and thread
through; early-return `Moved` when no version fields are set.

## M8. Worker-pool executor: adjacent hot atomics, per-op claiming,
notify-all storms

`include/tess/ops/phase_executor.h:355-382`: `next_offset_` and
`finished_operations_` are adjacent members (one cache line — they
ping-pong between workers), every operation costs two contended RMWs,
dispatch always `notify_all`s the full pool even for 2-op phases, and
completed workers `notify_all` on the shared cv. Existing bench
evidence: 256 one-tile ops take 23.7 us pooled vs 0.96 us serial.
Memory ordering itself was verified correct — this is purely
throughput. Fix: `alignas(64)` the two counters, claim small runs per
`fetch_add`, wake `min(workers, ops)` threads, last-worker-only
completion notify.

## M9. A* node state spans 4 arrays: up to 4 cache lines per node touch

`include/tess/path/path.h:540-548`: `generation_`, `state_`, `g_`,
`parent_` are separate allocations; every relaxation reads/writes the
same logical node across up to four lines in the hottest loop in the
library. An interleaved hot-node struct (`{gen, state, g}`, parent
optionally separate) halves-to-quarters the lines touched. Tradeoff
with the epoch-wrap bulk reset; measure via the path bench family.

## M10. `PathScratch::touched_` stores 8 bytes per reached node to
report a count

`include/tess/path/path.h:535-547`: verified that only
`DistanceFieldScratch::touched_` is ever iterated (dependency
construction, `field_product_cache.h:444`); `PathScratch::touched_` is
only `size()`d. A plain counter removes a store per newly-reached node
in both A* cores plus a world-sized allocation. (Keep the field
scratch's vector.)

## M11. Sparse LRU eviction is O(resident_capacity) per miss at budget;
sparse storage has zero bench coverage

`include/tess/storage/sparse_world.h:504-528`: eviction scans every
resident slot; a streaming workload at capacity pays O(loads x
capacity). The file itself notes an intrusive O(1) LRU list as the
replacement. Compounding it: no benchmark exercises `SparseWorld`
residency at all (every bench world is `AlwaysResidentWorld`; the
"sparse" bench names are scenario labels), so this scaling is ungated.
Fix: add a residency bench family (lookup, ensure_resident, eviction
under budget), then the intrusive list when it shows.

## Correctness findings (all LOW)

- C1. Unsaturated bounds arithmetic duplicated outside chunk_meta:
  `include/tess/ops/queued.h:715` (`dirty_axis_end`) and
  `include/tess/sim/render_delta.h:26` (`axis_end`) compute
  `origin + static_cast<int64_t>(extent)` without the saturation that
  `chunk_meta.h`'s `box_axis_end` applies for exactly this hazard
  (extent >= 2^63 flips negative; large origin+extent is signed
  overflow, UB). Reachable only with absurd caller-supplied `Box3`
  extents. Fix: share the saturating helper.
- C2. `Schedule::run_tick` leaves `in_run_` latched if a task callback
  throws (`include/tess/sim/schedule.h:231-248`): next tick
  assert-aborts in dev builds. Nothing documents a no-throw contract
  for schedule tasks (executors do document one). Fix: scope guard or
  documented contract.
- C3. Unweighted A* `f` arithmetic wraps rather than saturates
  (`include/tess/path/detail/astar.h:693`, `:618`), asymmetric with
  the weighted core's `saturating_add` (`:1049`). Needs ~2^32-tile
  distances to matter — not constructible today. Fix for symmetry.
- C4. Residency-fingerprint 64-bit collision is theoretically able to
  authorize stale field reuse (see M2's epoch alternative).

## LOW performance items

- `WeightedPortalSegmentCache` lookup and store are O(entries) linear
  scans (`include/tess/path/portal_segment_cache.h:142-156`; default
  budget 256) — the flat-hash pattern used by the route cache fits.
- `FieldProductCache` lookup/store/evict linear scans with full
  goals-vector compares (`field_product_cache.h:197,233,325`) — fine
  at tens of entries, index if it grows.
- Hand-rolled bit-loop popcount (`chunk_meta.h:40-47`) →
  `std::popcount`.
- `TileOccupancyIndex` hash is 3 chained `mix` calls (6 multiplies)
  and is recomputed per displaced entry during erase backward-shift
  (`ecs/adapter.h:317,243`).
- EnTT collect/apply re-`get` components already held by the view
  (`ecs/entt/entt_adapter.h:165-176,213-221`); a `PathState*` cached
  in the entry survives the sort.
- Weighted relaxation fetches neighbor entry cost before the
  `tentative_g < g` guard (`astar.h:1025`) — reorder if free.
- Sparse A* neighbor evaluation performs several directory probes
  (residency + offset + passability each re-probe;
  `path/node_index_space.h:72`) — a slot-returning single lookup can
  feed all three.
- Route-cache stores pay O(path_length) suffix-index inserts per
  cached Found route (`route_cache.h:286-294`) — by design; gate the
  suffix index for workloads that never suffix-hit.
- `parallel/` bench family (10 benches) still ungated; the stated
  precondition ("until CI baselines exist", `bench/CMakeLists.txt:373`)
  is now met — wire `thresholds/parallel.json`.
- Threshold gate targets run a single unreplicated sample while the
  10-rep data feeds only the non-gating artifact; pass
  `--benchmark_repetitions` to gate on the median.

## Verified clean (no action)

- Diagnostics disabled = zero cost: all `TESS_DIAG_*` expand to empty
  statements, arguments unevaluated (two independent confirmations).
- `WorkerPoolPhaseExecutor` join/publication ordering correct
  (release-RMW + acquire predicate + mutex handshake); epoch check
  handles late wakers; constructor/dispatch failure paths safe.
- ResultChannel prepare-upfront and drain semantics race-free;
  DeltaCollector coalesce table, truncation and version-chain
  semantics correct; open-addressed tables (occupancy, route cache,
  delta coalesce) all do correct backward-shift deletion.
- `UInt128`, `saturating_add` family, `box_axis_end`, assert macros
  under NDEBUG, tag_identity (no ICF flags in any preset), GPU
  descriptor static_asserts, `FixedStepAccumulator` edge cases: clean.
- Release bench flags genuine (`-O3 -DNDEBUG`, arm64); all gated
  families pass on this host at 8-63% of their Linux-calibrated caps.

## Suggested remediation order

1. H1 (bench integrity) + the two bench-infrastructure LOWs — restores
   trust in the gate before any perf work is measured against it.
2. M1 + M2 (+M3 spike behind a bench) — the batch-pathing tick is the
   product's stated hot path; these are its cheapest structural wins.
3. M5 + M6 + M7 — per-tick engine overhead with clear mechanical
   fixes.
4. M8-M11 + C1-C3 and the LOW list opportunistically alongside
   whichever subsystem is next touched.
